# 05 — The UI Layer

Source: `spotify-player/spotify_player/src/ui/` (7 files, ~2.9k LOC).
Stack: `ratatui` 0.x with the `crossterm` backend, optional `viuer` for cover art, optional `rustfft` for the audio bars.

## Render loop

The UI is a dedicated OS thread named `"ui"`, spawned from `main.rs`:

- `spotify_player/src/main.rs:194-197` — `std::thread::Builder::new().name("ui".to_string()).spawn(... ui::run(&state))`. There is no async/event-driven render — it is a plain loop.
- `spotify_player/src/ui/mod.rs:36` — entry point `pub fn run(state: &SharedState) -> Result<()>`.
- Tick model: dumb fixed-interval. Read the configured tick from `app_refresh_duration_in_ms` (default `32` ms ≈ 31 fps; see `spotify_player/src/config/mod.rs:332`), draw, then `std::thread::sleep(ui_refresh_duration)` (`ui/mod.rs:74`). No diff/dirty detection — every tick re-renders the whole frame. Ratatui's internal double-buffer is what saves the terminal from flicker.
- Terminal init in `init_ui` (`ui/mod.rs:79-91`): `crossterm::terminal::enable_raw_mode`, then `EnterAlternateScreen` + `EnableMouseCapture` via `crossterm::execute!`, wrap stdout in `CrosstermBackend`, build `ratatui::Terminal`, `terminal.clear()`.
- Teardown in `clean_up` (`ui/mod.rs:94-103`): `disable_raw_mode`, `LeaveAlternateScreen`, `DisableMouseCapture`, `terminal.show_cursor()`. Called from inside the loop when `ui.is_running` flips to false (`ui/mod.rs:47-50`), then `std::process::exit(0)` — note the explicit exit; teardown is *not* a `Drop` impl, so a panic in the render closure will leave the terminal in a bad state. Worth fixing in hibias (RAII guard).
- Resize handling: `terminal.size()` is sampled each tick and compared to `last_terminal_size`. On change the cover-image `ImageRenderInfo` is reset so the picture redraws (`ui/mod.rs:52-60`). No explicit `SIGWINCH` handler — ratatui/crossterm absorb it and the size delta is observed on the next tick.
- The render closure (`ui/mod.rs:62-71`) first paints a full-screen background `Block` styled with `ui.theme.app()`, then calls `render_application`. Render errors are logged via `tracing::error!` and swallowed — the loop never crashes.

## Top-level frame composition

`render_application` (`ui/mod.rs:106-119`) composes the screen in a fixed Z-order, each function consuming a `Rect` and returning the leftover area:

1. `playback::render_playback_window` carves off the playback strip (top or bottom).
2. `popup::render_shortcut_help_popup` carves off a help bar at the bottom *if* the user is mid-keychord.
3. `popup::render_popup` carves off the active modal (if any) and returns whether main-layout focus is still active.
4. `render_main_layout` matches on `ui.current_page().page_type()` and dispatches.

Order matters because of the cover image — `viuer` writes pixels directly to the terminal *outside* ratatui's buffer, so playback must render first to avoid being painted over by later widgets (see comment, `ui/mod.rs:107-111` + GitHub issue #498).

## Page model

A page is one entry in a stack: `UIState.history: Vec<PageState>` (`state/ui/mod.rs:33`). Pushing a page is `new_page` (`state/ui/mod.rs:63`); the topmost is "current". `PageState` is a discriminated union owning per-page UI sub-state (list scroll positions, focus enums, query input). Going "back" is just `history.pop()`.

`PageType` (`state/ui/page.rs:42-51`) is the matchable discriminant used by the render dispatcher.

| PageType      | PageState variant                      | Renderer (`ui/page.rs`)            | Sub-windows / focus                                              |
| ------------- | -------------------------------------- | ---------------------------------- | ---------------------------------------------------------------- |
| `Library`     | `Library { state: LibraryPageUIState }` | `render_library_page` :385         | Playlists / Albums / Artists; `LibraryFocusState`                |
| `Search`      | `Search { line_input, query, state }`  | `render_search_page` :34           | Tracks/Albums/Artists/Playlists/Shows/Episodes + input           |
| `Context`     | `Context { id, type, state }`          | `render_context_page` :255         | Branches on Playlist/Album/Artist/Tracks/Show; artist sub-pane   |
| `Browse`      | `Browse { state: BrowsePageUIState }`  | `render_browse_page` :512          | Two-state machine: Categories → Category playlists               |
| `Lyrics`      | `Lyrics { uri, track, artists }`       | `render_lyrics_page` :568          | Auto-scrolling timestamped lines                                 |
| `Queue`       | `Queue { scroll_offset }`              | `render_queue_page` :719           | Single table                                                     |
| `CommandHelp` | `CommandHelp { scroll_offset }`        | `render_commands_help_page` :649   | Three-column table of bound keys                                 |
| `Logs`        | `Logs { scroll_offset }`               | `render_logs_page` :1160           | Tail of `state.logs` ring buffer                                 |

Each `render_*_page` function is documented (`ui/page.rs:27-32`) to follow four steps: (1) read state, (2) build layout via `Layout::vertical/horizontal`, (3) construct widgets, (4) call `frame.render_widget`/`render_stateful_widget`. The pattern is rigorous — a good shape to adopt in hibias.

A subtle wart: each renderer takes `ui: &mut UIStateGuard`, immutably reads the page via `ui.current_page()` to build widgets, then re-borrows mutably via `ui.current_page_mut()` to get the `&mut ListState`/`&mut TableState` ratatui needs for `render_stateful_widget` (e.g. `ui/page.rs:204-217`, `ui/page.rs:485-509`). Each page does the let-else dance again. In hibias we can split `PageState` ownership from `UIStateGuard` to avoid this.

## Popup model

Popups are modal overlays driven by `ui.popup: Option<PopupState>` (`state/ui/popup.rs:19-39`). Only one popup is shown at a time; there is no z-stack.

| PopupState               | Renderer in `ui/popup.rs`        | Layout                                                           |
| ------------------------ | -------------------------------- | ---------------------------------------------------------------- |
| `Search { query }`       | inline at :65                    | Bottom 3-row strip; *non-focused* — main page still drives keys  |
| `ActionList`             | `render_list_popup` at :75       | Bottom list, height = `n_actions + 2`                            |
| `DeviceList`             | :90                              | Bottom list, height 5                                            |
| `ThemeList`              | :106                             | Bottom list, height 7                                            |
| `UserPlaylistList`       | :112                             | Search box + filtered list; height 10                            |
| `UserFollowedArtistList` | :167                             | Bottom list, height 7                                            |
| `UserSavedAlbumList`     | :180                             | Bottom list, height 7                                            |
| `ArtistList`             | :193                             | Bottom list, height 5                                            |
| `PlaylistCreate`         | :27                              | Bottom 3-row strip split 50/50: name + description inputs        |
| `ConfirmAction`          | :199                             | Bottom 3-row "y/n" prompt                                        |

`render_popup` (`ui/popup.rs:18`) returns `(remaining_rect, is_active)`. `is_active=false` means the popup steals focus, so the page renders in "inactive" style (no highlighted selection). Most popups carve off the bottom of the screen; none float over the content.

`render_shortcut_help_popup` (`ui/popup.rs:245`) is a separate, lower-priority overlay that appears whenever `ui.input_key_sequence` has buffered keys that prefix-match a multi-key chord. It enumerates remaining completions in a 3-column table at the bottom. Rendered before regular popups so popups can stack on top.

The list popups all share `render_list_popup` (`ui/popup.rs:220-242`) which builds the bordered block, calls `construct_list_widget`, and reaches back into `ui.popup.as_mut().unwrap().list_state_mut()` for the mutable `ListState`. That means each list popup variant carries its own `ListState` — see `state/ui/popup.rs:23-29`.

## Layout primitives

All layout uses ratatui's `Layout::vertical`/`Layout::horizontal` with `Constraint::{Length, Percentage, Ratio, Fill, Min}`. Two reusable helpers:

- `Orientation` enum (`ui/mod.rs:142-172`) — `Vertical` vs `Horizontal`. Auto-derived from terminal aspect ratio: `columns / rows > 2.3 ⇒ Horizontal` (`ui/mod.rs:151-159`). Stored on `UIState.orientation` and re-used by pages that have a "two-column when wide, stacked when tall" mode (library, search). The 2.3 cutoff accounts for terminal cells being ~2× taller than wide.
- `construct_and_render_block` (`ui/utils.rs:11-51`) — universal "render a titled bordered block, return the inner Rect" helper. Honors `BorderType::{Hidden, Plain, Rounded, Double, Thick}` from config; in `Hidden` mode it sets borders to `NONE` *after* computing the inner rect so paddings stay consistent.
- `construct_list_widget` / `render_list_window` / `render_table_window` (`ui/utils.rs:54-121`) — pair widget construction with state adjustment (`adjust_list_state` clamps stale selection indices when item counts shrink).

Layout pattern worth copying: every renderer threads `Rect` through with let-bindings rather than nested closures. The result is dead obvious to read and easy to refactor.

## Theming

`config::Theme` (`config/theme.rs:15-21`) has a name, a `Palette` (16 ANSI slots + optional fg/bg), and a `ComponentStyle` (per-semantic-role override). Colors deserialize from either named ANSI ("Cyan", "BrightRed") or hex ("#1e1e2e") — see `StyleColor::deserialize` at `config/theme.rs:412-453`.

The runtime API on `Theme` is a flat set of accessors that each return a `ratatui::style::Style`:

- Structural: `app()`, `border()`, `block_title()`, `selection(is_active)`, `secondary_row()`
- Playback: `playback_status()`, `playback_track()`, `playback_artists()`, `playback_album()`, `playback_genres()`, `playback_metadata()`, `playback_progress_bar()`, `playback_progress_bar_unfilled()`
- Tables/lists: `table_header()`, `current_playing()`, `like()`
- Pages: `page_desc()`, `playlist_desc()`, `lyrics_played()`, `lyrics_playing()`

Each accessor falls back to a hardcoded default if the theme didn't override it (e.g. `block_title` defaults to magenta — `config/theme.rs:198-202`). Themes from `theme.toml` are merged into the in-memory list at parse time (`config/theme.rs:147-167`); the *user* themes never override built-ins of the same name (first-write-wins). The active theme is selected by name from `app_config.theme` and copied onto `UIState.theme` at startup (`state/mod.rs:47-50`); `Theme` is `Clone` and lives on the UI state, so theming is just plain field reads on every render.

For hibias: this design is good — small flat accessor surface, default-with-override, decoupled from ratatui types. Worth copying. Consider exposing `Style` (not `Color`) as the unit, as spotify-player does, so individual roles can carry modifiers (bold/italic) too.

## Image rendering / album art

Gated behind the `image` Cargo feature (`Cargo.toml:46-47, 95-97`). Implementation in `ui/playback.rs:471-503`.

- Library: `viuer = "=0.9.2"`. `viuer::print` (`ui/playback.rs:485`) prints directly to the terminal (bypassing ratatui's buffer) at `(rect.x, rect.y)` with computed pixel dims.
- Protocol: `viuer` auto-detects support and uses **Kitty graphics**, **iTerm2 inline images**, or **Sixel** (the last is opt-in via the `sixel` feature, `Cargo.toml:96`). Falls back to half-block rendering when nothing else works.
- Caching: decoded images live in `data.caches.images`, keyed by URL (`ui/playback.rs:473`).
- Redraw control: `UIState.last_cover_image_render_info` (`state/ui/mod.rs:43-44`, struct at :17-23) caches `(url, render_area, rendered)`. The renderer compares against the current frame and only redraws when URL or area change (`ui/playback.rs:81-126`). This avoids flicker — `viuer` is *not* idempotent across frames.
- Buffer interaction: after drawing the image, the cells covering the image area are flagged with `set_skip(true)` (`ui/playback.rs:117-125`) so ratatui's buffer flush won't overwrite them. On size change or playback stop, `clear_area` zeroes the cells with theme-styled spaces *before* clearing `skip` flags (the comment at :115 warns that skip-cells are not cleared by buffer ops).
- Sizing: `cover_img_length` (rows), `cover_img_width` (cols), `cover_img_scale` (terminal-dependent multiplier — see issue #122 referenced at `ui/playback.rs:478-481`). All in `config/mod.rs:103-109`.

Caveat for hibias: the "render image outside the buffer + skip cells" trick is fragile. It dictates the entire Z-order of `render_application`. If we want hibias to be more flexible, we should either (a) render every frame's image fresh (pay the flicker), (b) use a buffer-respecting backend like `ratatui-image`, or (c) restrict art to a dedicated, stable region.

## Audio visualization

Implementation: `ui/streaming.rs` (409 lines), gated by the `streaming` feature. Two halves:

**Producer** (`VisualizationSink`, `ui/streaming.rs:111-275`): a `Sink` decorator wrapping librespot's audio backend. On every `write(packet)`:

1. Mix-down stereo to mono `f32` and push into `sample_buf: VecDeque<f32>` (`:191-199`).
2. While ≥ `FFT_SIZE` (1024) samples buffered, slide a Hann-windowed FFT over `HOP_SIZE` (128) ≈ 2.9 ms at 44.1 kHz (`:211-270`).
3. Compute magnitudes, fold into 128 log-spaced bands via `fill_log_bands` using *precomputed* bin ranges (`:282-305`). Apply a 3-tap `[0.25, 0.5, 0.25]` smoothing pass (`smooth_bands`, `:322-333`).
4. Acquire the shared `Mutex<VisBands>` (`:256-267`), apply wall-clock decay since last update (per-band `DECAY_FACTOR=0.985`, peak envelope `DECAY_FACTOR_PEAK=0.9985`), then `max(stored*decay, fresh)` so transients spike instantly. Update `peak_envelope` for normalization. Forward the unchanged packet to the inner sink.

All buffers (`fft_buf`, `magnitudes`, `new_bands`, `smooth_scratch`, `hann_window`) are pre-allocated in `new()` (`:147-169`) — zero allocation per hop. This level of care matters because the audio thread runs at ~344 hops/s.

**Consumer** (`render_audio_visualization`, `ui/streaming.rs:361-409`):

- Reads `state.vis_bands` (set up in `state/mod.rs:60-66` only when feature enabled).
- `try_lock`-style: takes the lock, *copies* the fixed-size `[f32; 128]` array out, drops the lock (`:370-379`). Crucial — the audio thread cannot afford to wait on the UI.
- Computes a render-side decay using `decay_for_elapsed(updated_at.elapsed())` (`:374`) so bars interpolate smoothly between writes.
- Subsamples to `rect.width` bars; height is `rect.height * 8` for ratatui's eighth-block sub-row resolution (`:380-394`); applies sqrt (gamma 0.5) curve to lift quiet signals; colors via `bar_color(t)` (cool blue → green → hot red, `:337-354`).
- Renders as `BarChart::default().bar_width(1).bar_gap(0)` (`:402-407`).

The visualization area is reserved by `split_rect_for_playback_window` (`ui/playback.rs:519-525`) only when `enable_audio_visualization && is_local_streaming_active()` — so nothing is drawn (and no rows are stolen) for Spotify Connect playback on a remote device.

For hibias: this whole subsystem is a textbook example of "shared FFT state via mutex with try-lock + per-frame decay" and is worth porting almost verbatim if we want a visualizer.

## The shared frame state pattern

The core idea: **acquire locks once at the top of the tick, render from the snapshot, drop locks before the sleep**.

- `ui/mod.rs:46` — outer block `let mut ui = state.ui.lock();` holds the UI mutex for the entire `terminal.draw(..)` call. The lock is released by the closing brace at `:72` *before* `std::thread::sleep` (`:74`).
- Inside the render, *player* and *data* state are accessed via short-lived RwLock read guards on each access:
  - `let player = state.player.read();` — `ui/playback.rs:30`
  - `let data = state.data.read();` — `ui/page.rs:50, :290, :393, :520, :576`
- `DataReadGuard` and `UIStateGuard` (`state/ui/mod.rs:8`) are `parking_lot::MutexGuard` aliases passed to helper renderers. Functions that need both guards take them as separate parameters rather than re-locking — see `render_track_table(ui, &data, ...)` (`ui/page.rs:938-1078`). This keeps lock scope explicit and prevents deadlocks.
- The `vis_bands` lock is held only long enough to memcpy the `[f32; 128]` out (`ui/streaming.rs:370-379`).

This is **not** a snapshot pattern in the conventional sense (no `Arc<Snapshot>` swap). It's per-tick blocking lock acquisition. It works because:

1. Page renderers don't await anything — they're synchronous and fast.
2. The data lock is `RwLock`, so the network thread's reads don't conflict.
3. The UI mutex is taken once at the top; only the network thread contends and only briefly.

Watch-out for hibias: the loop holds `ui.lock()` across a `terminal.draw()` that can issue blocking syscalls (image render via `viuer::print` is real I/O). On a stalled terminal, command threads that need `ui.lock()` will block too. If hibias cares about command latency under terminal stalls, prefer to: (a) snapshot UI state into a plain struct under the lock, drop the lock, render from the snapshot, then re-take the lock to write back any mutated `ListState`/`TableState`; or (b) move ratatui state out of `UIState` entirely and use ratatui's stateless widgets where possible.

## Patterns worth copying — and worth avoiding — in hibias

Copy:

- The four-step "get data → layout → widgets → render" structure for every page (`ui/page.rs:27-32`).
- `Orientation::from_size` + `Orientation::layout` — gives a clean "responsive" mode without per-page if-trees beyond one match (`ui/mod.rs:142-172`).
- `construct_and_render_block` for borders/title/inner-rect computation (`ui/utils.rs:11-51`). Universal.
- `Theme` as a flat set of style accessors, each with hardcoded defaults (`config/theme.rs:171-376`). Add roles incrementally.
- The visualizer producer/consumer split — pre-allocated buffers in the audio thread, snapshot copy in the render thread (`ui/streaming.rs`).
- `adjust_list_state`/`adjust_table_state` to clamp stale selections (`ui/utils.rs:80-110`). Otherwise off-by-one-on-shrink bugs are forever.
- Use `Constraint::Length(1)` rows for description bars; `Constraint::Fill(0)` to consume rest; `spacing(1)` on horizontal layouts for readable padding.

Avoid (or improve):

- Holding the UI mutex across the entire `terminal.draw` (see warning above).
- Cleaning up the terminal only via explicit `clean_up()` + `process::exit(0)`. A panic mid-render leaves the terminal in raw mode. Use a RAII guard.
- Coupling render Z-order to the cover-image hack. Either (a) commit to a buffer-respecting image backend (`ratatui-image`) or (b) constrain the image to one fixed pane and document the constraint.
- The "let-else twice" pattern (read page state, build widgets, then `let PageState::X { state, .. } = ui.current_page_mut() else { return };` to grab the `&mut ListState`) appears in nearly every renderer. Refactor to either pass `&mut ListState` in directly, or pull the mutable handle once at the start.
- Each popup variant carrying its own `ListState`. A single shared list cursor on the popup container would be simpler.
- Fixed `app_refresh_duration_in_ms` polling. For hibias, consider: tick on (a) actual key input, (b) a steady ~16 ms timer, and (c) any `state_changed` channel notification — woken from a `crossbeam` channel rather than `thread::sleep`.
